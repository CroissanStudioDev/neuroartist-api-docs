# Webhooks

Webhook — push-уведомление о завершении генерации. Gateway шлёт `POST` на ваш URL с подписью Svix-формата. Это альтернатива SSE для долгих и серверных пайплайнов.

## Когда использовать

| Хотите | Решение |
| --- | --- |
| Real-time UI в браузере | [SSE](/concepts/sse) |
| Серверный пайплайн без активного клиента | **Webhook** |
| Видео-генерация > 5 минут | **Webhook** |
| Гарантированная доставка | **Webhook** (есть retries) |

## Как подключить

<Steps>

### Подготовьте endpoint

Любой публичный HTTPS-URL, отдающий `200 OK` после успешной обработки. Любой 4xx/5xx ответ — gateway повторит доставку с экспоненциальным backoff.

### Получите webhook secret

```bash
curl https://api.neuroartist.ru/me/webhook-secret -b cookies.txt
```

```json
{ "secret": "wsec_base64_32_bytes_..." }
```

Если secret ещё не создан — `null`. Создайте через rotate:

```bash
curl -X POST https://api.neuroartist.ru/me/webhook-secret/rotate -b cookies.txt
```

```json
{ "newSecret": "wsec_..." }
```

<Callout type="warning">
  Secret показывается **один раз** при ротации. Сохраните в env вашего приложения. При компрометации — повторите rotate.
</Callout>

### Передайте `webhook` URL при submit

Через query-параметр:

```bash
curl -X POST 'https://api.neuroartist.ru/queue/fal-ai%2Fflux%2Fdev?webhook=https://your.app/api/na-webhook' \
  -H "Authorization: Bearer na_live_xxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"prompt":"…"}'
```

### Принимайте и валидируйте

См. ниже секцию [Валидация подписи](#валидация-подписи).

</Steps>

## Формат запроса

Headers:

```
POST /api/na-webhook HTTP/1.1
Content-Type: application/json
Webhook-Id: msg_abc123
Webhook-Timestamp: 1714050000
Webhook-Signature: v1,base64-encoded-hmac-sha256
User-Agent: NeuroartistGateway/1.0
```

Body — completed:

```json
{
  "type": "generation.completed",
  "requestId": "req_abc",
  "modelId": "fal-ai/flux/dev",
  "userId": "u_…",
  "createdAt": "2026-04-25T10:00:00Z",
  "completedAt": "2026-04-25T10:00:08Z",
  "result": {
    "images": [{ "url": "https://s3.cloud.ru/…", "width": 768, "height": 1024 }],
    "seed": 12345
  }
}
```

Body — failed:

```json
{
  "type": "generation.failed",
  "requestId": "req_abc",
  "modelId": "fal-ai/flux/dev",
  "createdAt": "…",
  "failedAt": "…",
  "error": { "code": "provider_error", "message": "…" }
}
```

URLs в `result` уже переписаны на наш S3 (если не указан `?passthrough=1`).

## Валидация подписи

Формат — Svix-style. Подпись считается над `{webhook_id}.{timestamp}.{body}`:

```js
import crypto from 'node:crypto'

function verify(req, secret) {
  const id = req.headers['webhook-id']
  const ts = req.headers['webhook-timestamp']
  const sig = req.headers['webhook-signature']  // 'v1,base64'

  // Защита от replay
  if (Math.abs(Date.now() / 1000 - Number(ts)) > 300) return false

  const payload = `${id}.${ts}.${req.rawBody}`
  // secret хранится в формате 'wsec_base64' — декодируем base64-тело
  const key = Buffer.from(secret.replace(/^wsec_/, ''), 'base64')
  const expected = 'v1,' + crypto
    .createHmac('sha256', key)
    .update(payload)
    .digest('base64')

  // Сравниваем по signatures-list (через запятую — для ротации)
  return sig.split(' ').some(s =>
    crypto.timingSafeEqual(Buffer.from(s), Buffer.from(expected))
  )
}
```

<Callout type="error">
  Никогда не сравнивайте подписи через `===`. Это уязвимо к timing-атакам. Используйте `crypto.timingSafeEqual` (Node), `hmac.compare_digest` (Python), `hash_equals` (PHP).
</Callout>

## Идемпотентность

Webhook может прийти повторно (ретрай после network blip). Дедупайте по `webhook-id` или `requestId`:

```js
const seen = await redis.set(`wh:${webhookId}`, '1', 'NX', 'EX', 86400)
if (seen === null) return res.status(200).end()
```

## Ретраи

Если ваш endpoint вернул `4xx`/`5xx`/таймаут (> 30 сек), gateway повторяет доставку:

| Попытка | Задержка |
| --- | --- |
| 2 | 1 минута |
| 3 | 5 минут |
| 4 | 30 минут |
| 5 | 2 часа |
| 6 | 5 часов |

После 6-й попытки доставка прекращается, событие помечается `failed`. Все попытки логируются в `/me/activity`.

## Poller vs Webhook

Внутри gateway есть два режима доведения генерации до terminal-состояния:

| Режим | Когда |
| --- | --- |
| `poller` (default) | Gateway сам опрашивает провайдер каждые ~2с |
| `webhook` | Provider шлёт callback на gateway, gateway — на ваш URL |

Это **внутренняя деталь** — для клиента разницы нет, оба пути дают одинаковый итоговый webhook на ваш URL. Различие важно только при self-host gateway: `poller` — без публичного URL, `webhook` — публичный URL обязателен.

## Входящие webhooks (то, что gateway принимает)

Это ручки **для провайдеров и платежей**, не для вас:

| Эндпоинт | От кого | Защита |
| --- | --- | --- |
| `POST /webhooks/{providerId}` | fal.ai, replicate, azure | ED25519-подпись провайдера |
| `POST /webhooks/yookassa/notify` | YooKassa | IP-whitelist + HMAC |

В клиентский код ходить туда не нужно — они работают server-to-server между gateway и upstream-ами.
