# SSE-прогресс

Async-генерация (`POST /queue/{modelId}`) возвращает `requestId` сразу. Чтобы получить terminal-результат не опросом, а push-стримом — подписывайтесь на SSE.

## Два эндпоинта

| Эндпоинт | Что отдаёт |
| --- | --- |
| `GET /queue/{modelId}/requests/{requestId}/status/stream` | Лента статуса от провайдера (raw fal-формат) |
| `GET /queue/{modelId}/requests/{requestId}/progress/stream` | Универсальная лента gateway-а с lifecycle-стадиями |

Используйте `progress/stream` — он унифицирован и переживает рестарт backend-а (late-join).

## Жизненный цикл

```
queued → in_progress → uploading_assets → reconciling → completed
                                                       ↘  failed
```

| Стадия | Когда |
| --- | --- |
| `queued` | Submit принят, ждём слот у провайдера |
| `in_progress` | Провайдер начал генерацию |
| `uploading_assets` | Артефакты выгружаются в наш S3 |
| `reconciling` | Списание кредитов корректируется по фактическому usage |
| `completed` | Готово, можно забирать результат |
| `failed` | Провайдер вернул error либо пайплайн упал |

## Формат события

Frame:

```
data: {"stage":"in_progress","requestId":"req_abc","timestamp":"2026-04-25T10:00:01Z","payload":{...}}

```

Минимальный shape — `stage` + `requestId` + `timestamp`. Дополнительные поля зависят от стадии (`logs`, `progress` от провайдера, `assetIds` после re-host, `error` при failure).

При `completed`:

```json
{
  "stage": "completed",
  "requestId": "req_abc",
  "timestamp": "…",
  "result": {
    "images": [{ "url": "https://s3.cloud.ru/…", "width": 768, "height": 1024 }],
    "seed": 12345
  }
}
```

При `failed`:

```json
{
  "stage": "failed",
  "requestId": "req_abc",
  "timestamp": "…",
  "error": { "code": "provider_error", "message": "…" }
}
```

## Heartbeat

Каждые `SSE_HEARTBEAT_MS` миллисекунд (по умолчанию 15 000) gateway шлёт comment-frame:

```
: ping
```

Это держит соединение живым через reverse-proxy с idle-timeout-ом. Клиенту обрабатывать его не нужно — `EventSource` игнорирует comment-frames автоматически.

## Лимит соединений на пользователя

`SSE_MAX_CONNECTIONS_PER_USER` (по умолчанию 5) — максимум одновременных SSE-стримов с одного аккаунта. 6-й вернёт:

```
HTTP/1.1 429 Too Many Requests
Retry-After: 60

{"code":"too_many_sse_connections","message":"Too many concurrent SSE connections"}
```

Слот освобождается при закрытии стрима либо через TTL 1 час (self-healing на случай зависшего клиента).

<Callout type="warning">
  Не открывайте новый SSE на каждый рендер. EventSource прекрасно переживает переключение вкладок, но течь будет — старые подключения не закрываются автоматически.
</Callout>

## Late-join (присоединение позже)

`progress/stream` поддерживает подключение **в любой момент** жизненного цикла, в том числе **после** terminal-стадии. Gateway сразу отдаст последнее известное событие, потом закроет стрим.

Это удобно для UI, который монтируется уже после submit (например, после refresh страницы).

## Когда лучше webhook вместо SSE

| Вместо SSE используйте webhook когда | Почему |
| --- | --- |
| Долгие генерации (видео > 5 минут) | SSE-соединение через мобильную сеть отвалится |
| Pipeline без активного клиента | Сервер не может «не закрыть вкладку» |
| Нужна гарантия доставки | Webhook ретраится, SSE — нет |

См. [Webhooks](/concepts/webhooks).

## Регрессии и устранение

| Симптом | Причина |
| --- | --- |
| Стрим обрывается через ~60 сек | Idle-timeout reverse-proxy | Снизьте `SSE_HEARTBEAT_MS` либо подправьте proxy timeout |
| `429 too_many_sse_connections` | Утечка стримов на клиенте | Закрывайте `EventSource` после terminal-события |
| Получили только heartbeat-ы, нет событий | Провайдер не публикует прогресс | Это норма — terminal придёт `completed`/`failed` |
| После рестарта backend нет terminal | Poller-mode без webhook-ов | См. [архитектура: poller vs webhook](/concepts/webhooks#poller-vs-webhook) |
