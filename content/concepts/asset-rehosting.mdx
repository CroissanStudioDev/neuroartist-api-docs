# Хранение результатов

После завершения генерации gateway автоматически переписывает ссылки на артефакты — из CDN провайдера (где они живут несколько часов) в наш S3. Клиенту не нужно ничего менять.

## Зачем

CDN провайдеров — временный storage. Через ~24 часа ссылка возвращает 404, а часто и быстрее. У вас в БД может оказаться битая URL.

Наш S3 (Cloud.ru) хранит файлы пока вы не удалите их через `/me/activity`. Это даёт:

- Долговременные ссылки в вашем коде.
- Контроль удаления — GDPR-friendly, можно стереть всё для конкретного юзера.
- Стабильный CDN с одинаковыми latency-ями для разных регионов.

## Как работает

При завершении generation pipeline:

```
┌──────────┐   ┌─────────┐   ┌──────────┐   ┌─────────────┐
│ provider │ → │ gateway │ → │   S3     │ → │ rewritten   │
│   CDN    │   │ download│   │ upload   │   │ URLs to you │
└──────────┘   └─────────┘   └──────────┘   └─────────────┘
```

1. Gateway скачивает файл с CDN провайдера (max `ASSET_MAX_BYTES` = 500 MB).
2. Загружает в наш S3 bucket с прозрачным `content-type`.
3. Переписывает все URL-ы в payload результата на S3-ссылки.
4. Возвращает вам уже переписанный payload.

Этот процесс — синхронный относительно вашей подписки на `/queue/.../result` или webhook. На SSE он обозначается стадией `uploading_assets`.

## Где это срабатывает

- `POST /run/{modelId}` — синхронный режим, переписывание перед ответом.
- `POST /queue/{modelId}` → `GET .../result` — переписывание перед выдачей результата.
- Webhook от провайдера → ваш webhook — переписывание перед forwarding.

## Что переписывается

- `images[].url`, `image.url`
- `video.url`
- `audio.url`
- Все вложенные URL-ы из вложенных payload-shapes конкретных моделей (определяется per-provider).

`seed`, `timings`, `metadata` — оставляются как есть.

## Опциональный bypass — passthrough

Если вы хотите получить **исходные** ссылки от провайдера (для отладки, для интеграций с инструментами провайдера), добавьте `?passthrough=1` к ручке результата:

```bash
curl 'https://api.neuroartist.ru/queue/fal-ai%2Fflux%2Fdev/requests/req_abc?passthrough=1' \
  -H "Authorization: Bearer na_live_xxxxxxxxxxxxxxxx"
```

В ответе будут оригинальные `https://fal.media/...` URL-ы. Подразумевайте, что они умрут через несколько часов.

То же работает для `/run/{modelId}?passthrough=1`.

## Что если upload упал

Любая ошибка S3 (5xx, таймаут, превышение `ASSET_MAX_BYTES`) — graceful fallback: gateway оставляет CDN-URL провайдера в результате. Запрос НЕ упадёт, генерация считается успешной.

В `/me/activity/{requestId}` будет запись об asset со статусом `failed`. Вы получите working ссылку — но она протухнет на стороне провайдера через несколько часов, скачайте её к себе пораньше.

## Размер и MIME

| Параметр | Значение по умолчанию |
| --- | --- |
| `ASSET_MAX_BYTES` | 500 MB (на один артефакт) |
| Поддерживаемые MIME | `image/*`, `video/mp4`, `video/quicktime`, `video/webm`, `audio/*` |

Артефакты сверх лимита не загружаются — будет fallback на CDN провайдера.

## Жизненный цикл

```
pending → uploading → ready
                    ↘ failed
```

Видно через `/me/activity/{requestId}` → `assets[].status`.

## Удаление

```bash
# По одному реквесту — удалит все assets реквеста
curl -X DELETE https://api.neuroartist.ru/me/activity/{requestId}/assets \
  -H "Authorization: Bearer na_live_xxxxxxxxxxxxxxxx"
```

<Callout type="warning">
  Удаление необратимо. После удаления — ссылка возвращает 404, восстановления нет. Скачайте к себе, если нужны архивы.
</Callout>

## Отдельный случай — input uploads

Для `image-to-image` и `video-to-video` моделей вы заливаете input через `POST /me/uploads`. Эти файлы хранятся **в том же S3**, со своим `assetId`. Удаляются по той же модели — через `/me/activity` или после reaper TTL (если настроен).
